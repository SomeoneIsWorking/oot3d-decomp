# OoT3D Scene Lighting — ZSI EnvLightSettings Layout

Reverse-engineered 2026-06-23 by static analysis + empirical validation.

## Summary

OoT3D scene headers (ZSI files) embed environment lighting settings via command
`0x0F` (the same command type as N64 OoT). The OoT3D entry format is **28 bytes**
per setting (vs N64's 22 bytes), with the color/direction fields first and the fog
distance floats at the end.

---

## OoT3D EnvLightSettings entry (28 bytes = 0x1C)

```
+0x00  u8[3]  ambientColor   RGB, scale 0..255
+0x03  s8[3]  light0Dir      unit vector × 127 (XYZ)
+0x06  u8[3]  light0Color    RGB, scale 0..255
+0x09  s8[3]  light1Dir      unit vector × 127 (XYZ)
+0x0c  u8[3]  light1Color    RGB, scale 0..255
+0x0f  u8     pad            (purpose unknown; sometimes non-zero)
+0x10  f32 LE fogEnd         world-units distance (typically 20000–40000)
+0x14  f32 LE drawDistance   world-units distance (varies; render/fog start dist)
+0x18  u16 LE blendFog       bits[0..9] = fogStart, bits[10..15] = blendRate/4
+0x1a  u8[2]  unknown        not consumed by known game paths
```

**Important:** Entry 0 in every scene has a different binary layout (it appears to
be a per-scene metadata record or defaults blob). Entries 1..N-1 are the actual
per-time-of-day (or event) lighting settings consumed by the runtime interpolator.
Entry 0 is preserved in the JSON but should not be used as a lighting setting.

## Differences from N64 OoT EnvLightSettings (22 bytes)

| Field        | N64 (22 B) offset | OoT3D (28 B) offset | Notes |
|--------------|-------------------|----------------------|-------|
| ambientColor | +0x00 u8[3]       | +0x00 u8[3]          | same position |
| light0Dir    | +0x03 s8[3]       | +0x03 s8[3]          | same position |
| light0Color  | +0x06 u8[3]       | +0x06 u8[3]          | same position |
| light1Dir    | +0x09 s8[3]       | +0x09 s8[3]          | same position |
| light1Color  | +0x0c u8[3]       | +0x0c u8[3]          | same position |
| fogColor     | +0x0f u8[3]       | **missing / split**  | see below |
| fogNear      | +0x12 s16         | +0x18 u16 (packed)   | moved after floats |
| fogFar       | +0x14 s16         | merged into +0x18    | N/A in OoT3D? |
| (new) pad    | —                 | +0x0f u8             | replaces fogColor[0]? |
| (new) fogEnd | —                 | +0x10 f32            | exact world-unit fog end |
| (new) drawDist | —              | +0x14 f32            | world-unit draw distance |

The noclip project's `OcarinaOfTime3D/zsi.ts` describes a **different** 28-byte
format with floats at +0x00 and +0x04 — that layout applies to N64 ZSI files
embedded inside the 3DS game, NOT to native 3DS ZSI files. Do not apply it to
OoT3D romfs ZSI data.

## Key observations

- **Colored ambient**: OoT3D uses fully RGB ambient, not greyscale. Kokiri Forest
  day entries have `ambient = (79..119, 72, 72)` — a slight reddish tint. Market
  Night entries have `ambient = (10..40, 176, 89)` — a teal/green tint.
- **Light directions are partly scene-constant**: For a given scene, `light0Dir[0]`
  (the X component) is often constant across all time-of-day entries (e.g. +72 for
  Kokiri noon entries). Only the Y and Z components vary per setting.
- **fogEnd is often scene-constant**: Kokiri = 32000, Market Night = 20000. Only
  `drawDistance` varies significantly across entries.

## Validation against live oracle values

| Scene | Setting | Field | Static extraction | Live oracle (N64 for comparison) |
|-------|---------|-------|-------------------|----------------------------------|
| Kokiri Forest | E8/E9 (noon) | ambient[0] | 79/255 = 0.310 | ~0.314 (N64: 80/255) |
| Market Night | E3/E7 (deep night) | ambient[0] | 10/255 = 0.039 | ~0.098 (N64: 25/255) |

OoT3D and N64 ambient values differ because Grezzo redesigned the lighting model.
The N64 `lightparams` REPL reads N64 OTR values; OoT3D ZSI has different artistic
choices (colored ambient, different magnitudes).

## Data file

`data/scene_lighting.json` — generated by `tools/scene_lighting.py`. Covers all
114 OoT3D scene headers. Format:

```json
{
  "scene_name": [
    {
      "ambient": [r, g, b],
      "light0": { "dir": [x, y, z], "color": [r, g, b] },
      "light1": { "dir": [x, y, z], "color": [r, g, b] },
      "fog_end": float,
      "draw_dist": float,
      "fog_start": int,
      "blend_rate": int
    }
  ]
}
```

Entry 0 per scene should be treated as a metadata record, not a usable lighting
setting. Entries 1..N-1 are the per-time-of-day settings in interpolation order.

## ZSI command list parsing notes

- ZSI magic = `ZSI\x01` (OoT3D native), not `ZSI\x00` or `ZSI ` (N64).
- Commands start at byte offset 16 (after 16-byte header: magic + scene name).
- Each command = 8 bytes: `cmd1` (big-endian u32) + `cmd2` (little-endian u32).
  - `cmd1 >> 24` = command type (0x0F = env settings).
  - `(cmd1 >> 16) & 0xFF` = entry count.
  - `cmd2` = file offset to data block.
- Command type `0x14` = END marker.
